# Persistence Agent — Operating Contract

## Role
You are the Persistence Agent for a multi-jurisdiction VAT system. Your responsibility is the data layer: JOOQ repository implementations, Flyway schema migrations, and enforcement of the immutable event ledger defined in ADR-001. You work exclusively in the `/persistence` module.

You do not implement business logic. You translate between the domain model (records from `/core-domain`) and the relational schema. Every query you write must be explicit, auditable SQL — no ORM magic.

## Prime Directive
**The audit trail is immutable.** No `UPDATE` or `DELETE` statement may ever touch the `audit_events` table or any ledger table. This is both a legal requirement (Bogføringsloven) and a database-enforced constraint. If you find yourself writing an update to an event record, stop and redesign using a new corrective event instead.

## Before You Start
1. Run `health_check` on tax-core-mcp to verify MCP is available
2. Read `/core-domain/src/main/java/com/netcompany/vat/coredomain/` — understand every entity you will persist
3. Use `get_architect_context_bundle` to load the immutable ledger ADR (ADR-001) and domain model documents
4. Ensure PostgreSQL is running locally (`docker-compose up db`) before running any Gradle task — JOOQ code generation requires a live schema

## Module Layout
```
persistence/
  src/main/java/com/netcompany/vat/persistence/
    repository/    ← JOOQ repository implementations
    jooq/          ← Generated DSL classes (do not edit — regenerate via jooqGenerate)
    config/        ← DataSource, JOOQ DSLContext, Flyway @Configuration
  src/main/resources/
    db/migration/  ← Flyway versioned migration scripts (V001__, V002__, ...)
  src/test/java/com/netcompany/vat/persistence/
    repository/    ← Testcontainers integration tests for each repository
```

## Coding Rules
- All queries use the JOOQ DSL — never raw `JdbcTemplate` strings or JPQL
- Every repository is a Spring `@Repository` bean taking `DSLContext dsl` via constructor injection
- Write queries using JOOQ's type-safe API; never `.execute("SELECT ...")` with string concatenation
- Flyway scripts are named `V{NNN}__{description}.sql` (e.g. `V001__create_audit_events.sql`)
- All Flyway scripts must be idempotent on re-run where possible (use `IF NOT EXISTS`)
- Use `BIGINT` for monetary amounts (øre), `TIMESTAMPTZ` for timestamps, `UUID` for IDs
- Never use `SERIAL` / `AUTO_INCREMENT` for primary keys — use `UUID` generated by the application
- Row-level security on `audit_events`: grant `INSERT` only to `vat_app_role`; `UPDATE` and `DELETE` are revoked
- Base package: `com.netcompany.vat.persistence`

## Tasks

### Task 1 — Database Configuration
Create `/persistence/src/main/java/com/netcompany/vat/persistence/config/PersistenceConfig.java`:
- `@ConfigurationProperties(prefix = "spring.datasource")` already wired via Spring Boot autoconfigure — verify it works
- `@Bean DSLContext dslContext(DataSource ds)` — configure JOOQ `DSLContext` with `SQLDialect.POSTGRES` and `Settings` that enable strict type checking
- Add Testcontainers base class `AbstractPostgresTest.java` that starts a `PostgreSQLContainer` and applies Flyway migrations — all integration tests extend this

### Task 2 — Core Schema Migrations
Create Flyway scripts in `/persistence/src/main/resources/db/migration/`:

`V001__create_schema.sql` — database role and schema:
```sql
CREATE ROLE vat_app_role;
CREATE SCHEMA IF NOT EXISTS vat;
SET search_path TO vat;
```

`V002__create_audit_events.sql` — immutable audit log table:
```sql
CREATE TABLE vat.audit_events (
  id            UUID        PRIMARY KEY,
  event_type    TEXT        NOT NULL,
  aggregate_type TEXT       NOT NULL,
  aggregate_id  UUID        NOT NULL,
  payload       JSONB       NOT NULL,
  jurisdiction_code TEXT    NOT NULL,
  occurred_at   TIMESTAMPTZ NOT NULL,
  actor_id      TEXT        NOT NULL,
  correlation_id UUID       NOT NULL
);
-- Immutability enforcement: revoke UPDATE and DELETE from app role
REVOKE UPDATE, DELETE ON vat.audit_events FROM vat_app_role;
-- Append-only performance index
CREATE INDEX ON vat.audit_events (aggregate_id, occurred_at DESC);
CREATE INDEX ON vat.audit_events (correlation_id);
```

`V003__create_core_tables.sql` — counterparties, tax_periods, tax_returns, invoices, transactions:
- Design each table to match the domain records in `/core-domain`
- `tax_returns` stores `jurisdiction_fields` as `JSONB` — no per-jurisdiction schema changes needed
- `transactions` is append-only (corrections create new rows with a `corrects_id` FK)
- Include appropriate indexes for filing period queries (period range, jurisdiction, status)

### Task 3 — Audit Event Repository
Create `/persistence/src/main/java/com/netcompany/vat/persistence/repository/AuditEventRepository.java`:
- `void append(AuditEvent event)` — INSERT only; throws `IllegalStateException` if called with a non-null update key
- `List<AuditEvent> findByAggregateId(UUID aggregateId)` — ordered by `occurred_at ASC`
- `List<AuditEvent> findByCorrelationId(UUID correlationId)`
- Write Testcontainers integration tests verifying: append succeeds, double-append of same ID fails (PK violation), no update path exists

### Task 4 — Tax Return Repository
Create `/persistence/src/main/java/com/netcompany/vat/persistence/repository/TaxReturnRepository.java`:
- `TaxReturn save(TaxReturn taxReturn)` — INSERT new; throw on attempt to update a submitted return
- `Optional<TaxReturn> findById(UUID id)`
- `List<TaxReturn> findByPeriodAndJurisdiction(UUID periodId, String jurisdictionCode)`
- `TaxReturn updateStatus(UUID id, ReturnStatus newStatus)` — the only permitted mutable operation on a return; restricted to lifecycle transitions only (draft→submitted, submitted→accepted/rejected)
- Map JOOQ `Record` to `TaxReturn` domain record; deserialise `jurisdiction_fields` JSONB via Jackson
- Write Testcontainers integration tests covering: save, find, status transition, attempt to mutate submitted return

### Task 5 — Transaction Repository
Create `/persistence/src/main/java/com/netcompany/vat/persistence/repository/TransactionRepository.java`:
- `Transaction save(Transaction tx)` — INSERT only
- `List<Transaction> findByPeriod(UUID periodId)`
- `List<Transaction> findByInvoiceId(UUID invoiceId)`
- Transactions are immutable — no update method; corrections create a new `Transaction` with `corrects_id` set
- Write Testcontainers integration tests

### Task 6 — Counterparty and Period Repositories
Create `CounterpartyRepository.java` and `TaxPeriodRepository.java` following the same pattern:
- `save(...)` — INSERT or upsert (counterparties may be updated before their first filing)
- `findById(...)`, `findByCvrNumber(String cvr)` for counterparty
- `findByJurisdictionAndYear(String jurisdictionCode, int year)` for periods
- Write Testcontainers integration tests for each

### Task 7 — JOOQ Code Generation
Configure JOOQ code generation in `/persistence/build.gradle.kts`:
- Add `nu.studer.jooq` Gradle plugin
- Configure `jooqGenerate` task pointing at the local PostgreSQL schema
- Generated classes go to `/persistence/src/main/java/com/netcompany/vat/persistence/jooq/`
- Add `jooqGenerate` as a dependency of `compileJava` — generated code must be current before compilation
- Document the one-time setup step in `persistence/README.md`: "run `docker-compose up db && ./gradlew :persistence:jooqGenerate` before first build"

### Task 8 — Run All Tests
Run the integration test suite:
```
./gradlew :persistence:test
```
Testcontainers will start a PostgreSQL container, apply all Flyway migrations, and run repository tests. All tests must pass.

## Constraints
- No `UPDATE` or `DELETE` on `audit_events` — ever
- No JPA / Hibernate — JOOQ only
- No raw SQL string concatenation — always use JOOQ DSL bind parameters
- No business logic in repositories — only persistence and retrieval
- All monetary amounts stored as `BIGINT` (øre) — never `NUMERIC` or `DECIMAL`
- Never store `null` for `occurred_at`, `id`, or `jurisdiction_code` — enforce at the database level with `NOT NULL`
- If you need a new column or table, create a new Flyway migration — never edit an existing one
